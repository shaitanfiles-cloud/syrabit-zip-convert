# Workspace — Syrabit.ai

## Overview

Syrabit.ai is an AI-powered educational platform designed for students in Assam, India (AHSEC Class 11/12 and Degree). It offers localized learning resources across 55 subjects, leveraging AI for content generation, syllabus management, and SEO. The platform aims to provide personalized education, enhance content delivery through chapter-level RAG chunks, and ensure accessible, engaging, and high-quality educational content via a robust admin panel. The core mission is to deliver an affordable, AI-first learning experience with significant market potential in the regional education sector.

## User Preferences

I prefer iterative development with clear communication on major changes. I value detailed explanations for complex features and architectural decisions. Please ensure that the development process prioritizes modularity and maintainability.

## System Architecture

The project utilizes a pnpm workspace monorepo, featuring a React + Vite frontend and a FastAPI Python backend.

**Frontend Architecture:**
- **UI/UX:** React, Vite, React Router, Tailwind CSS, mobile-first responsive design, light-only theme.
- **Admin Panel:** Comprehensive content management system for editing, CMS, blog publishing, SEO management, QA review, and system intelligence, including custom alert sound uploads and an audio trimming component.
- **Bot-Aware Pre-Rendering:** `BotRenderMiddleware` provides pre-rendered HTML to search engines and manages `robots.txt`, `sitemap.xml`, and `sitemap-index.xml`.
- **Bot Discovery Infrastructure:** Includes RSS feeds, machine-readable manifests (`/llms.txt`, `/llms-full.txt`), AI plugin discovery (`/.well-known/ai-plugin.json`), and IndexNow integration for instant URL indexing.
- **PWA:** Multi-cache service worker for offline capabilities and performance.
- **SEO Optimization:** Single SEO landing pages with clean URLs, SERP preview modals, deduplicated heading IDs, `PageMeta` for all pages, JSON-LD, programmatic SEO engine, premium keyword expansion, topic keyword index, and `SpeakableSpecification`.
- **Analytics:** Multi-source analytics (Cloudflare, GA4, server-side, JS-tracked) with Core Web Vitals.
- **Bilingual Support:** Content available in English and Assamese, independently stored and accessible via UI toggles.
- **Content Display:** Library page with subject cards, lesson pages with blog-style layout, reading progress, and sticky TOC.
- **Onboarding:** Streamlined processes for DEGREE and AHSEC/SEBA students.

**Backend Architecture:**
- **Modular Design:** App factory pattern with shared modules and route modules, enforcing explicit API surfaces via `__all__`.
- **AI Integration:** On-demand generation and management of chapter embeddings. Utilizes Vertex AI / Gemini for embeddings, translation, vision analysis, content enhancement, quality scoring, and SEO meta generation. Single-LLM pipeline for concise responses.
- **Content Pipeline:** Parallel generation of notes, MCQs, and flashcards using `asyncio.gather`. Features a detailed content generation prompt for exam-ready study notes.
- **Content Feedback Loop:** Auto-detection of thin chapters, auto-healing with version history, and quality gates.
- **Admin Analytics:** Dashboard displaying RAG telemetry, chat latency, user counts, content heatmaps, and a historical alert log with real-time notifications.
- **PYQ HTML Replica:** Processes PYQ PDFs via Gemini Vision OCR for SEO-optimized, RAG-indexed HTML.
- **Syllabus Embedder:** Generates 768-dimensional chapter/topic embeddings stored in Cloudflare Vectorize.
- **Monetization:** Supports free, starter, and pro plans with credit-based usage.
- **Security:** ASGI-native `SecurityHeadersMiddleware`, prompt safety, spoofed bot UA monitoring, and automated IP blocking. OpenAPI schema is suppressed in production for security.
- **Privacy:** Tracks DPDP Act consent.
- **Performance Optimizations:** Bounded content caching, efficient JWT decoding, thread pooling, MongoDB compound indexes, hierarchy caching, AsyncOpenAI client pooling, parallelized chat pre-processing, and throttled LLM health probes. Achieves sub-1s chat latency for English queries.
- **Educational Browser Backend:** Infrastructure for an in-app educational browser with grounded AI chat, including domain allowlisting, content fetching, and kid-safe content filtering.

## GitHub Actions supply-chain hardening

- **SHA-pinned actions (Task #883):** every `uses:` reference under `.github/workflows/*.yml` is pinned to a full 40-char commit SHA with a trailing `# vX.Y.Z` tag comment. Float tags like `actions/checkout@v4` are forbidden because a compromise of the upstream tag — as with `tj-actions/changed-files` in 2025 — would silently exfiltrate repo secrets on the next push.
- **Self-enforcing pin gate (Task #895):** `.github/workflows/pinned-actions-check.yml` greps every `uses:` line on each PR and fails the merge unless the ref matches `@[a-f0-9]{40}`. Wired as a required check on master branch protection. Dependabot's weekly bumps update the SHA and trailing comment in lockstep.
- **Least-privilege `GITHUB_TOKEN` (Task #905):** every workflow declares an explicit top-level `permissions:` block (or per-job blocks) granting only the scopes its jobs actually need — `contents: read` by default. Without an explicit block the repo default (`contents: write`) applies, which would let a hypothetically-compromised pinned action push commits, open PRs, or write packages from inside any job. Today only two workflows escalate: `backend-tests.yml` (`pull-requests: write` for the PR-coverage comment, downgraded to read-only on forked PRs) and `trustpilot-aggregate-refresh.yml` (`contents: write` for its auto-commit of the Trustpilot cache file). The convention is documented in the supply-chain header comment block at the top of every workflow file; new workflows MUST set `permissions:`.
- **Workflow-security linter gate (Task #906):** `.github/workflows/workflow-security-scan.yml` runs Trail of Bits' `zizmor` against every file in `.github/workflows/` on each PR to `master` and fails the merge on any finding of severity HIGH or above. Required-check name is **`zizmor — workflow-security scan (HIGH+ blocks merge)`**. Catches the classes of GitHub-Actions footguns the SHA-pin gate is silent on: `pull_request_target` / `workflow_run` token exposure to forked-PR code, `${{ ... }}` script injection into `run:` blocks, cache-poisoning patterns, `actions/checkout` leaving the `GITHUB_TOKEN` in `.git/config` for a later `upload-artifact` step to glob in (`artipacked`), unredacted secrets in logs, and known-vulnerable third-party action versions. zizmor itself is hash-pinned (`zizmor==1.24.1` with sha256s of both the manylinux wheel and the sdist) and installed via `pip install --require-hashes --no-deps`, so the gate's own supply chain is locked. The whole workflow tree is currently clean at default severity (zero suppressions in `.github/zizmor.yml`); informational findings still print in the run log for triage but do not block. To bump zizmor: update the version + both sha256s on the pinned-requirements heredoc inside `workflow-security-scan.yml` (look them up at `https://pypi.org/project/zizmor/<version>/#files`).

## Unified Log Explorer (Task #944)

- **Goal:** single admin pane that ingests logs from three producers — Cloudflare Pages (frontend), the edge-proxy Worker, and the Railway-hosted FastAPI backend — into one Mongo collection (`unified_logs`) so an on-call admin can pivot/search/export across the entire request path.
- **Storage:** `unified_logs_dao.py` owns the collection. TTL via `expire_at` field + index `expireAfterSeconds=0`; supporting indexes are `(timestamp DESC)`, `(source, timestamp DESC)`, sparse `correlation_id`, `(level, timestamp DESC)`, sparse `(status, timestamp DESC)`. All inserts go through `insert_logs` which clips long fields, drops malformed rows, honours a caller-supplied `_id` for idempotent upserts, and treats `BulkWriteError` duplicate-key (E11000) as "already ingested" rather than a hard error.
- **Sampling/safeguards:** `should_keep_request()` always retains 4xx/5xx and ≥1500 ms-slow requests regardless of sample rate; remainder kept with probability `EDGE_LOG_SAMPLE_RATE` (env, applied at edge worker shipper) and `BACKEND_LOG_SAMPLE_RATE` (process-local, applied by the in-process backend shipper). Hard caps: `UNIFIED_LOGS_MAX_INGEST_BATCH=500` records per ingest call, `UNIFIED_LOGS_TTL_DAYS=14` default. A runtime kill-switch (`UNIFIED_LOGS_PAUSE=1` env OR `set_runtime_pause(True)` set via the admin Pause endpoint) drops every record with `paused=True` and counts them as `dropped` so the producer sees an honest tally instead of silent loss.
- **Routes (all under `/api/admin/logs`, admin auth required except ingest):**
  - `POST /api/logs/ingest` — worker shipper endpoint, gated by `X-Logs-Ingest-Token` (constant-time compare against the value persisted in `api_config` collection). Token is rotated by the admin via `POST /rotate-token` which returns the plaintext **once** and writes only an audit breadcrumb (no token in logs).
  - `GET /api/admin/logs` — filter/search/paginate (`sources`, `levels`, `search`, `status`, `min_duration_ms`, `route`, `country`, `colo`, `cache`, `ray_id`, `correlation_id`, `since`, `until`, `limit`, `cursor`).
  - `GET /api/admin/logs/trace/{correlation_id}` — pivots from a single record to every other record sharing the same `correlation_id`, regardless of source (the "trace this request" shortcut).
  - `GET /api/admin/logs/export?fmt=csv|ndjson` — streaming export, same filter shape as the list endpoint.
  - `GET /api/admin/logs/status` — full health pane (paused?, TTL, sampler stats, CF cursor, per-source counts).
  - `GET /api/admin/logs/sources/stats` — lightweight per-source counts + last-seen timestamp (used by the live-tail header so it doesn't re-fetch the full status payload every poll).
  - `POST /api/admin/logs/cf/pull` — manual one-shot trigger of the Cloudflare GraphQL pull (used after rotating the CF token, or to debug without waiting for the next loop tick).
  - `POST /api/admin/logs/pause` / `POST /api/admin/logs/resume` — admin kill-switch.
  - `DELETE /api/admin/logs?sources=…` — destructive purge, optionally scoped by source.
- **Cloudflare pull loop:** Background task (`_unified_logs_cf_pull_loop`) queries `httpRequestsAdaptiveGroups` every `CF_PULL_INTERVAL_S` (default 60s). Cursor is a single Mongo doc in `job_locks` (`_id="unified_logs_cf_pull"`) so a process restart resumes from `until` instead of re-pulling the whole window. The CF query intentionally uses **top-level `count`** (not `sum.requests`) and omits `rayName` because `httpRequestsAdaptiveGroups` is the **aggregated** dataset — neither field exists in its schema (`rayName` only appears on the per-event Logpush dataset, which requires Enterprise). Each CF row gets a deterministic `_id = "cf_" + sha1(minute|method|path|status|colo|cache)` so a retry inside the same minute window collapses onto the same Mongo doc instead of double-counting. On consecutive failures the loop applies exponential backoff (`base * 2^n`) capped at 30 min with ±25% jitter, and resets to the base interval on the next successful tick.
- **Edge worker shipper:** `workers/edge-proxy/src/log-shipper.ts` collects per-request log entries in an in-memory ring buffer and flushes them via `ctx.waitUntil(fetch(BACKEND_INGEST_URL, …))` so the user-facing response is never blocked. Authenticates with the rotated `X-Logs-Ingest-Token` from worker secret storage. Wrapped in `index.ts` so every fetch handler emits an entry with method, path, status, duration, country, colo, cache status, and `cf-ray` (which becomes both `ray_id` and `correlation_id` for that record — the only producer that has a per-request ray identifier). On low-traffic windows where the size + age thresholds don't fire, the first record into an empty buffer schedules a single `ctx.waitUntil(setTimeout(flush))` (default 5s, env `EDGE_LOG_DEFERRED_FLUSH_MS`) with a single-flight guard so a trickle of one request a minute still gets shipped within seconds instead of sitting in isolate memory until eviction.
- **Frontend:** `artifacts/syrabit/src/components/admin/AdminLogsExplorer.jsx` — registered in `AdminPage`. Filter bar covers all server-side filters; live-tail polls `GET /api/admin/logs` every 5s with the newest seen timestamp as `since`; export buttons hit `/api/admin/logs/export?fmt=csv|ndjson` via `adminLogsDownloadExport` (axios blob fetch with the admin Bearer token in `Authorization`, then a synthetic `<a download>` click) so the download works in environments that don't have an admin cookie; falls back to `window.open(adminLogsExportUrl(...))` if the blob path errors. Each row has an expand-chevron that reveals the **full pretty-printed JSON payload** (with a "Copy JSON" button) plus a one-click **Copy correlation id** button next to the trace shortcut so an on-call can paste the cid into Slack/Linear without opening the row. A collapsible **"Ingest safeguards"** card surfaces TTL/retention, backend & edge sample rates, max ingest batch, CF pull cadence, and ingest-token state — these values are **read-only / env-driven by design** (mutating them at runtime is followup #950 below) so the operator always knows what guardrails are active.
- **CF idempotency key (important):** the deterministic `_id` for a Cloudflare row MUST include **every** dimension the GraphQL `httpRequestsAdaptiveGroups` query groups by — currently `minute|method|path|host|country|colo|edgeStatus|originStatus|cacheStatus`. Any dimension added to the `_CF_QUERY` `dimensions` block must also be appended to the idempotency key in `normalize_cf_http_request_row()`, otherwise two legitimately distinct buckets (e.g. same path/status from two different countries in the same minute) would collide on the same `_id` and the second one would be silently swallowed by the E11000-tolerance in `insert_logs()`. `tests/test_unified_logs_cf_pull.py::test_normalize_cf_row_idempotency_key_includes_all_grouping_dims` is the regression guard.
- **Tests (63 total, all green):** `tests/test_unified_logs_dao.py` (18) — normalisation, sampling, indexes; `tests/test_admin_logs_route.py` (13) — auth gates, filters, exports, ingest token, pause; `tests/test_unified_logs_cf_pull.py` (6) — CF GraphQL row mapping + dedupe-key collision guard, cursor advance, no-config short-circuit. Worker: `workers/edge-proxy/tests/log-shipper.test.ts` (20) — batching, sampling, token rotation, waitUntil non-blocking. Frontend: `AdminLogsExplorer.test.jsx` (6) — SSR header/filter chips/columns + a live-DOM test that asserts row-expansion shows the full JSON and the copy-correlation-id button calls `navigator.clipboard.writeText`.

## External Dependencies

- **Databases:** PostgreSQL, MongoDB, Cloudflare D1.
- **Authentication:** Supabase, JWT helpers, Google OAuth.
- **Caching:** Cloudflare AI Gateway (upstream LLM cache), Cloudflare edge worker KV bindings (rate limiting, bot HTML cache).
- **LLM Providers:** Cerebras, Groq, OpenRouter (general English chat); Cerebras (qwen-235b) and Gemini 2.5 Flash (admin content generation — notes / important questions / PYQ); Gemini for vision and embeddings; **Sarvam is gated to Assamese-only paths**, with two distinct routing contracts:
  1. **Assamese translation (Gemini main + Sarvam polish)** — substantive English→Assamese translations call `vertex_services.translate(target_lang="as")` first; if Gemini returns ≥ `_POLISH_MIN_LEN` (80) chars, the output is then sent through Sarvam's `/v1/chat/completions` with `model=sarvam-m` and a native-Assamese editor system prompt to polish phrasing/script consistency (1.8s budget). Per-fragment sanitiser splices below the length gate skip the polish round-trip. Polish failure or `sarvam_llm_client is None` degrades to the un-polished Gemini output. Implemented in `routes/ai_chat.py::_assamese_translate_gemini_main_sarvam_polish` and used by both `_make_assamese_translate_callable` and the instant fast-path. Sarvam's `/translate` endpoint is no longer the primary translation path.
  2. **Assamese chat response (Sarvam main + Gemini fallback, sequential)** — `call_llm_api_stream` for `response_lang == "as"` runs in two phases instead of a hedged parallel race: **Phase 1** races every key in `_SARVAM_PROVIDERS` for sarvam-m; the first key to emit a token wins and stragglers are cancelled. **Phase 2** is reached *only* if every Sarvam key fails before first token — it then streams a single Gemini 2.5 Flash attempt as fallback. This guarantees Sarvam-MAIN: Gemini cannot "steal" first-token from Sarvam due to network jitter. If Phase 2 also has no key configured, the SSE stream surfaces a user-facing "temporarily unavailable" error rather than silently hanging.
  - Sarvam is intentionally absent from `_LLM_PROVIDERS`, `_LLM_PROVIDERS_CHAT`, `_LLM_PROVIDERS_CONTENT`, `_SLM_SLOT_CANDIDATES`, and `_CONTENT_SLOT_CANDIDATES` — it lives in its own `_SARVAM_PROVIDERS` list and is reachable only through the two Indic-mode routes above. All LLM traffic is routed through Cloudflare AI Gateway. Contract pinned by `tests/test_assamese_translation_polish_contract.py` (8 tests covering Gemini-first call order, polish skip on short fragments, polish-failure degradation, Phase 1 win, Phase 2 fallback, and no-Gemini error path).
