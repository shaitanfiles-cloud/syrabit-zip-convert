Phase 1 — Chat & Content Generation (Biggest Impact)
What changes:

llm.py: Promote workers-ai from emergency fallback to primary provider. Remove Groq, Cerebras, OpenRouter, OpenAI, and Vertex from the _LLM_PROVIDERS pool
vertex_chat.py: Retire the entire file or stub it to delegate to Workers AI
config.py: Remove all provider key loading (_GROQ_KEY, _GEMINI_KEY, _CEREBRAS_KEY, _OPENROUTER_KEY, _OPENAI_KEY)
providers/workers_ai.py: Promote from fallback client to primary, add a new chat_primary() path that calls the edge worker's Workers AI binding directly — no more /api/ai/fallback/ emergency path, just a clean /api/ai/chat Workers AI path
workers/edge-proxy/src/index.ts: Update the edge worker to use env.AI.run("@cf/meta/llama-3.3-70b-instruct-fp8-fast", ...) with streaming support
Env vars removed: GROQ_API_KEY, GEMINI_API_KEY, CEREBRAS_API_KEY, OPENROUTER_API_KEY, OPENAI_API_KEY, VERTEX_PROJECT_ID, VERTEX_SERVICE_ACCOUNT

Files to update: llm.py, config.py, deps.py, vertex_chat.py, server.py (startup probe), workers/edge-proxy/src/index.ts

Phase 2 — Embeddings
What changes:

vertex_services.py: Replace _embed_one() (currently POSTs to generativelanguage.googleapis.com for gemini-embedding-001) with a call to the edge worker's /api/ai/embed → env.AI.run("@cf/baai/bge-base-en-v1.5", { text })
The output dimension stays 768 — BGE-base produces 768-dim vectors, same as gemini-embedding-001, so Cloudflare Vectorize index needs no schema change
retrievers/vertex.py: Update retriever to use the new embed path
Note: BGE-m3 (@cf/baai/bge-m3) is the better choice if you want multilingual quality (handles Assamese text natively). Tradeoff: it outputs 1024-dim, which would require re-indexing Vectorize.

Recommendation: Start with bge-base-en-v1.5 (768-dim, zero Vectorize migration) and upgrade to bge-m3 in a later pass.

Phase 3 — Vision, OCR, PDF Extraction
What changes:

vertex_services.analyze_image(), ocr_image(), extract_from_document(): Replace Gemini Vision API calls with edge worker routing to env.AI.run("@cf/meta/llama-3.2-11b-vision-instruct", { messages: [{ role, content: [{ type: "image_url", image_url: { url: dataUrl } }, { type: "text", text: prompt }] }] })
routes/pyq.py: PYQ PDF OCR uses extract_from_document — same update, but PDFs need to be converted to images page-by-page before passing to the vision model (Llama Vision takes images, not raw PDFs unlike Gemini)
Add a small PDF-to-image utility using Pillow + pdf2image (already have Pillow in requirements.txt)
Files to update: vertex_services.py, routes/pyq.py

Phase 4 — Translation & Assamese Language
What changes:

vertex_services.translate(): Replace Gemini translate call with edge worker → env.AI.run("@cf/meta/m2m100-1.2b", { text, source_lang: "eng_Latn", target_lang: "asm_Beng" })
llm.py Sarvam pool (_SARVAM_PROVIDERS): For Assamese chat responses, replace sarvam-m / sarvam-105b with @cf/meta/llama-3.3-70b-instruct-fp8-fast with a system prompt in Assamese instructing the model to reply in Assamese. Workers AI's Llama 3.3 70B handles Assamese reasonably well for conversational responses.
routes/ai_chat.py: Simplify _assamese_translate_gemini_main_sarvam_polish() — remove the Gemini main + Sarvam polish two-step, replace with a single M2M100 call
⚠️ Quality caveat: M2M100 at 1.2B parameters will produce grammatically correct but sometimes stilted Assamese compared to Sarvam's specialized model. If translation quality is critical for you, Sarvam for translation should be kept and only Sarvam's LLM chat replaced. Flag this as a conscious tradeoff.

Env vars removed: SARVAM_API_KEY, SARVAM_TRANSLATE_KEY

Phase 5 — Speech (STT / TTS)
What changes:

STT: Replace Sarvam STT with @cf/openai/whisper-large-v3-turbo — pass audio bytes to edge worker → env.AI.run("@cf/openai/whisper-large-v3-turbo", { audio: base64bytes })
TTS: @cf/myshell-ai/melotts-en is English-only. For Assamese TTS, there is currently no Workers AI model. Options:
Keep Sarvam TTS only for Assamese audio output (1 remaining Sarvam API key)
Disable Assamese TTS and serve text-only for now
Use a Web Speech API approach on the frontend for TTS (no backend needed)
Files to update: deps.py (sarvam_client setup), routes/ai_chat.py audio handling

Phase 6 — Edge Worker Upgrade
The existing workers/edge-proxy/src/index.ts already has a WORKERS_AI_FALLBACK_SECRET path. It needs to be promoted:

Add [ai] binding to wrangler.toml (one line: binding = "AI")
Add unified /api/ai/* handler: chat, embed, vision, translate, stt, tts
All Workers AI calls go through env.AI.run(model, input) — no external HTTP, no API keys in transit, billed directly to your Cloudflare account
Remove the fallback secret mechanism entirely — this becomes the primary path, not a fallback
