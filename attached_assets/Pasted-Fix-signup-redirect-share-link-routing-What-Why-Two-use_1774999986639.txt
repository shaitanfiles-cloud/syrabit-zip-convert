Fix signup redirect & share link routing
What & Why
Two user-facing bugs need fixing:

Signup redirect fails — After successful signup, users are redirected to /onboarding via navigate('/onboarding'). However, the OnboardingPage has a useEffect that checks if (!user) navigate('/login'). Because React state updates (setUser) from the AuthContext haven't propagated by the time OnboardingPage's effect runs, the user is bounced back to /login instead of staying on /onboarding.

Share link doesn't open the correct content — The share button on a SubjectPage calls createShare() with window.location.pathname as the subject_path. The backend redirector at /r/{code} redirects the visitor to that path (e.g., /ahsec/class-11/physics?ref=code). However, this path matches the SEO route /:board/:classSlug/:subjectSlug → SubjectLandingPage, which is a different page from the SubjectPage (at /subject/:subjectId) where the share button lives. The user lands on the wrong page. Additionally, if the SubjectPage was reached via /subject/:subjectId, the share link redirects to that path which does work — but the content card link should consistently open the right content.

Done looks like
After signup, users reliably land on the /onboarding page without being bounced to /login
Shared links open the correct content page that was originally shared
Both flows work on first try without requiring a page refresh
Out of scope
Redesigning the share system or referral tracking
Adding share buttons to other pages
Changing the onboarding flow itself
Tasks
Fix signup→onboarding race condition — In OnboardingPage, replace the bare useEffect redirect with a proper guard pattern that accounts for loading state from AuthContext, so the redirect to /login only fires after auth state is confirmed as null (not just initially unset). Alternatively, wrap the /onboarding route with a lightweight guard or add the loading check to the existing effect.
Fix share link target path — Ensure the subject_path stored in the share document maps to a route that renders the actual shared content. Either update createShare in SubjectPage to pass a canonical path that resolves to the correct page, or update the redirect logic to construct a path that matches the SubjectPage route (/subject/:subjectId).
