Enterprise Pipeline Full Audit
What & Why
Perform a complete audit of every stage in the enterprise content + AI pipeline. The goal is to identify what is fully wired, what is broken, what is missing, and what is blocked — then fix or wire each gap. This covers 10 distinct pipeline levels as described by the product owner.

Pipeline Levels Under Audit
1. Lesson Generate Level
Generate chapter notes from chapter name + syllabus content via AI
Backend must produce structured notes and persist them to the chapter document in MongoDB
2. Agentic Generate Level
Before generating, check if chapter notes already exist — if yes, skip notes generation
If notes do not exist, trigger generation first
After notes exist, auto-generate mark-wise important questions: 1 mark, 2 mark, 3 mark, 5 mark, and 10 mark
All question sets must be saved under the chapter
3. Flashcard / Memory Tricks Level
Generate flashcards containing: memory tricks, shortcuts to remember syllabus content, mindmaps (text-based), and memory hacks
These must be generated per chapter and attached to the chapter document
4. Chapter Update Level
After all generation (notes, questions, flashcards) completes, the chapter document in MongoDB must be updated atomically with all generated content
Frontend chapter/lesson view must display each section: notes, mark-wise questions, flashcards
5. SEO Generate Topic Level
Each SEO topic generated must be explicitly linked (by reference ID) to its parent chapter and to its content card
Verify the link is two-way: from topic page → chapter, and from chapter → topic pages list
6. SEO Polish Level
Each generated SEO topic page must go through a polish/refinement pass (rewrite, enrich, deduplicate)
Polished pages must remain linked to their chapter and content card — linkage must not break after polish
7. Publish as Blog Level
Polished SEO pages must be publishable as blog posts
Blog publish must set correct canonical URL, publish date, structured data (JSON-LD), and breadcrumb linking back through topic → chapter → subject
8. Content Card Level
Content cards must display per subject/chapter and show: card name, linked lesson, linked SEO topics, mark-wise questions count, flashcard count
Cards must be linked to the content they describe (chapter document + generated artifacts)
9. AI Chat Pipeline Level
This is the most critical level — audit every sub-step:

Embedding search order: query must first search topic embeddings → chapter embeddings → subject content card embeddings (in that order, with fallback)
Exact topic match: when a topic is matched, AI must scrape/retrieve from the matched content card and lesson content (not generic web)
Web search fallback: only after content card + lesson content search yields insufficient context should web search be triggered
Data source display: the chat response source panel must show only: content card name + lesson name (no raw URLs, no subject IDs, no internal keys)
Breadcrumb source trail: display topic → chapter → subject as a source path in the chat UI alongside each answer
10. Personalized CMS Level — /cms/{user_id}/{slug} (Paid, Scrapers BLOCKED)
Route /cms/{user_id}/{slug} must exist and be gated behind the paid tier
The scraper pipeline feeding this CMS is currently BLOCKED — audit why (missing API key, broken endpoint, permission error, or not implemented) and document the exact blocker
Do not implement the scraper; only identify and document the blocker clearly in code comments or a status flag the admin panel can display
Done looks like
Each of the 10 pipeline levels above is audited: documented as Working / Broken / Missing / Blocked
All broken or missing wiring is fixed so the end-to-end flow works: syllabus → notes → questions → flashcards → chapter update → SEO topics linked → SEO polish linked → blog publish → content card display → AI chat with correct source display
AI chat always shows "Content Card Name / Lesson Name" as the source, never raw IDs
The personalized CMS route exists and the scraper blocker is surfaced clearly in the admin panel
Out of scope
Building a new scraper for the personalized CMS (blocked — only identify and surface the blocker)
Redesigning the UI beyond functional display of generated content
Adding new AI providers
Tasks
Audit & fix Lesson Generate Level — Verify the API endpoint that generates chapter notes from chapter name + syllabus content; ensure notes are saved to MongoDB and the frontend lesson view renders them correctly.

Audit & fix Agentic Generate Level — Add skip-if-exists check before notes generation; wire the mark-wise question generation (1/2/3/5/10 mark) to run after notes are confirmed present; save all question sets to the chapter document.

Audit & fix Flashcard / Memory Tricks Level — Verify the flashcard generation prompt produces memory tricks, shortcuts, mindmaps, and memory hacks; ensure output is saved to the chapter document.

Audit & fix Chapter Update Level — Ensure the chapter document is updated atomically after all generation steps; verify the frontend displays notes, mark-wise questions, and flashcards in the lesson/learn view.

Audit & fix SEO Topic → Chapter → Content Card Linking — Verify bi-directional references between generated SEO topic pages, their parent chapter, and their content card; fix any missing links.

Audit & fix SEO Polish Level — Confirm the polish/refinement pass runs on each generated SEO topic page and that linkage to chapter and content card survives the polish step.

Audit & fix Blog Publish Level — Verify published blog posts carry canonical URL, structured JSON-LD data, publish date, and breadcrumb back to chapter and subject.

Audit & fix Content Card Level — Verify content cards display: card name, lesson link, SEO topic list, question counts by mark, and flashcard count.

Audit & fix AI Chat Pipeline — Audit embedding search order (topic → chapter → subject), confirm exact-topic content card retrieval fires before web search fallback, and fix the source display to show only content card name + lesson name with a topic → chapter → subject breadcrumb trail.

Audit Personalized CMS Route & Scraper Blocker — Verify /cms/{user_id}/{slug} route exists and is paid-gated; identify the exact reason the scraper is blocked and surface the status in the admin panel with a clear label (e.g. "Scraper: BLOCKED — [reason]").
