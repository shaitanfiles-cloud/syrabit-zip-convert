#35 - Fix Redis AI Response Caching
What & Why
The Redis AI cache has a high-severity key collision bug: cache keys are built from the query text only, ignoring subject, board, and RAG context. This means a Physics answer can be served to a Chemistry question with the same wording. Additionally, streaming cache hits break the streaming UX by returning the full answer at once, and the in-memory TTL cache is redundant when Redis is available.

Done looks like
Cache keys include subject ID, board ID, and relevant context identifiers so responses are never mixed across different curriculum contexts
When a cache hit occurs during a streaming response, the frontend receives tokens progressively (or an equivalent chunked response) rather than a single instant dump
Redundant in-memory _ai_response_cache is removed or disabled when Redis is confirmed available, reducing server RAM usage
Common casual queries ("hello", "help") are cached at a short TTL to avoid hitting the LLM for every repeated greeting
Out of scope
Changing the Redis provider (Upstash stays)
Caching non-AI endpoints
Changing cache TTL durations substantially
Tasks
Fix cache key collision — Update _cache_key to hash the combination of query text + subject_id + board_id (and optionally a hash of the RAG context snippet) so keys are context-scoped.
Fix streaming cache hit UX — When a cache hit is returned inside chat_stream, chunk the cached text into small token-sized pieces and yield them with a short delay so the frontend streaming experience is preserved.
Remove redundant in-memory cache — When Redis is available and connected, skip populating _ai_response_cache to avoid double-caching the same data in RAM.
Cache casual/greeting queries — Remove the hard exclusion for casual queries; cache them with a short TTL (e.g. 300s) so repeated greetings do not hit the LLM.
