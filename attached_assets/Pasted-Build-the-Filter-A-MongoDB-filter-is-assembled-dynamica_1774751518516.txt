Build the Filter
A MongoDB filter is assembled dynamically:

status = "published"   ← always
if subject_id is known:
    → lookup subject slug from subjects collection
    → add subject_slug = <slug> to filter
if only subject_name is given (no ID):
    → add subject_name regex filter
THEN add keyword OR-regex across 3 fields:
    content   $regex "keyword1|keyword2|keyword3"
    topic_title   $regex "keyword1|..."
    title         $regex "keyword1|..."
So the filter is: published + subject scoped (if available) + keyword hits in content or title.

Step 3 — Two parallel MongoDB queries
Both run at the same time via asyncio.gather:

Query	Collection	Limit	What it fetches
seo_task	seo_pages	6 docs	content, topic_title, chapter_title, page_type, slug
ch_task	chapters	4 docs	title, content, subject_id
Total: max 10 documents fetched per request.

Step 4 — Priority Ranking
From the 6 seo_pages:

Notes pages first (page_type == "notes") — highest educational value
Then any other seo_page type
Top 3 selected
From the 4 chapters:

Top 2 selected
So at most 5 content blocks go forward.

Step 5 — _extract_relevant_sections (per document)
Each selected document is trimmed intelligently — not just content[:N]:

Split content into paragraphs
Score each paragraph:
  +1 per keyword found
  +0.5 if paragraph is a header (starts with # or **)
Sort by score descending
Pick highest-scored paragraphs + their neighbours (±1 paragraph context)
until max_chars budget is reached
seo_pages get max 3500 chars each
chapters get max 2000 chars each
Step 6 — Assemble the Card
Selected blocks are joined with headers and separators:

[Content: Photosynthesis Notes]
## Light Reactions
...relevant paragraphs...
---
[Content: Chapter 3 — Plant Biology]
...relevant paragraphs...
---
[Chapter: Cell Biology]
...relevant section...
This full string is the content card — it goes directly into the AI system prompt as grounding context.

Why it's fast (10–50 ms)
No external API calls — pure MongoDB I/O
Scoped by subject first (index hit, not full scan)
Hard-limited to 10 docs max
_extract_relevant_sections runs in-process (no I/O)
No caching needed — it's already near-instant
