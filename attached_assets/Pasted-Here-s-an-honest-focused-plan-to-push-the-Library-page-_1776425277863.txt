Here's an honest, focused plan to push the Library page toward a 100/100 mobile Lighthouse score. I'll group it by impact so the biggest wins come first.

Phase 1 — Kill render-blocking + shrink critical bytes (biggest LCP/TBT wins)
Inline critical CSS for the library hero (header, search, first-row cards) and defer the rest of the stylesheet with media="print" onload.
Self-host fonts instead of Google Fonts (fonts.googleapis.com + fonts.gstatic.com cost a TLS handshake + render-blocking CSS). Ship Space Grotesk .woff2 from /fonts/, font-display: swap, and remove the Google Fonts links.
Trim the entry chunk: move react-helmet-async, sonner, posthog, the cache-cleanup IIFE, the emergent badge nuker, and PWA install prompt to fully idle/post-LCP loaders.
Drop unused JS on the library route: audit dist/stats.html, prune anything pulled in only by other pages (admin, MDX, sandpack remnants).
Pre-render /library HTML at build time (static prerender via vite-plugin-prerender or a build script that hits the live API and writes a hydrated HTML snapshot). This single change is what gets most React SPAs from ~70 to 95+ on mobile.
Phase 2 — LCP < 2s on Slow 4G
Identify the LCP element with WebPageTest filmstrip; on /library it's almost certainly the H1 / first card.
Preload the slim library JSON as a <link rel="preload" as="fetch" crossorigin> in index.html so the bundle request starts before the JS even parses.
Server-push or 103 Early Hints for the slim bundle and main CSS via Cloudflare.
Preconnect to the API origin and image CDN (already partly done — verify crossorigin is set correctly).
Compress and re-export /logo.webp (5 KB → ~1.5 KB), /opengraph.jpg (39 KB → ~12 KB AVIF), /yoga-cover-notext.webp (203 KB → ~40 KB AVIF).
Phase 3 — TBT < 100 ms
Virtualize the subject grid with @tanstack/react-virtual once filtered list > 20.
Web Worker for the filter pass so search never blocks the main thread (overkill for most cases but required for a perfect score on Moto G4).
Strip lucide-react entirely on this route — replace the ~10 icons with raw inline SVG components (saves ~8–12 KB and a chunk request).
Remove react-query from the critical path if not strictly needed; use a tiny custom fetch + cache for the slim bundle.
Disable PostHog session recording on /library (or load PostHog only after first interaction).
Phase 4 — Zero CLS
Reserve fixed heights for the sticky header (e.g. min-h-[140px]), filter row (min-h-[44px]), and each card (aspect-ratio or min-h).
Skeleton must be byte-for-byte the same dimensions as the loaded grid (already partly done — finish matching the language toggle width and card internals).
Reserve space for the lazy CMS sections with min-h placeholders so they don't shove content when they hydrate.
Phase 5 — Image audit pass
Serve subject thumbnails through an image CDN (Cloudflare Images or /cdn-cgi/image/) with width, format=auto, quality=75, and a real srcset.
Convert all PNG icons in /icons/ to optimized WebP/AVIF.
Add fetchpriority="high" to the LCP image (only one!) and fetchpriority="low" to below-the-fold thumbs.
Phase 6 — Network & caching
Brotli 11 at the edge for HTML/JS/CSS (Cloudflare default is fine; verify Content-Encoding: br).
Cache-Control: public, max-age=31536000, immutable for every hashed asset (already on /assets/*; extend to /icons/* hashed variants).
stale-while-revalidate on /api/content/library-bundle* so repeat visits paint instantly from cache.
HTTP/3 + 0-RTT — already on Cloudflare; just confirm enabled.
Phase 7 — Verification loop
Run mobile Lighthouse 5× on the Preview URL, take the median, fix the next worst opportunity.
Run PageSpeed Insights field data weekly until CrUX shows green for LCP/INP/CLS.
Realistic expectation
Hitting a perfect 100 on mobile is genuinely hard with a React SPA + third-party scripts (PostHog, AdSense, Cloudflare beacon, emergent.sh). The single change that makes it possible is Phase 1 step 5 — static prerender of /library combined with Phase 3 step 15 — kicking PostHog and AdSense off the critical path. Without those two, the realistic ceiling is ~92–96.

PR A (this task): Phases 1–4 — prerender, self-host fonts, virtualize, fix CLS. Should land 90+.
PR B (follow-up): Phases 5–6 — image CDN, third-party script gating. Pushes to 98–100.
