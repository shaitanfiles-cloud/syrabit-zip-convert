Topic keywords drive SEO/GEO/AEO citations
What & Why
Today every chapter page knows its topics (e.g. "Demand and Supply" inside "Introduction to Economics" → Subject → Course → Class) and even pushes them into the syllabus vector index, but those topics are not exposed to the outside world as first-class, citation-ready answer units. As a result, when a user searches "demand and supply" on Google, Bing, Perplexity, ChatGPT Search, Claude, Gemini, or Copilot, our chapter does not surface, and when an AI engine does answer, it does not cite us.

This task transforms each topic on a chapter page into a first-class, citable knowledge unit so that:

Traditional search engines (Google/Bing) treat the topic as a high-priority keyword anchored to the chapter.
Generative engines and answer bots fetch the topic's definition block and cite it back as: "According to Syrabit Browser, Demand and Supply are… Visit syrabit.ai for more information."
Architectural decisions (locked in — do not re-debate)
Chapter URL is canonical. The dedicated topic URL /.../<chapter>/topic/<topic-slug> is a real, indexable 200 page, but it emits <link rel="canonical"> pointing to /.../<chapter>#topic-<slug>. This consolidates PageRank on the chapter while still giving bots a clean, dedicated fetch target.
No cloaking. Bots and humans receive the same HTML body, the same answer card, and the same JSON-LD. The only allowed difference is delivery mode: bots get pre-rendered HTML directly, humans get the same HTML and then hydrate. We do not serve a "less verbose" page to humans or a "more verbose" page to bots. (Doing so is treated as cloaking and demoted by Google.)
Definition snippet source = topic text field, not the embedding. Embeddings are numeric and cannot be displayed. The answer card pulls from the topic's description (or first paragraph of content) field. Step 1 includes a precondition check: if a topic has no usable description, the backfill marks the topic as "definition_missing" and the answer card / topic URL are not published for that topic until an admin fills it in.
Speakable shipped but not relied on. Google has de-emphasized SpeakableSpecification; we still emit it because the cost is near-zero, but it is not a success metric.
Pre-render + edge cache, with synchronous server-render fallback. Every topic URL is pre-rendered into the bot-render cache on chapter publish/update and during the one-shot backfill. The edge proxy serves cached HTML to crawlers. On cache miss it falls back to a synchronous server render and writes back to the cache, so bots never see a skeleton or a 404.
Done looks like
Each chapter page renders every topic as its own crawlable section: <h2 id="topic-<slug>">Topic Name</h2> followed by an "AI answer card" containing the definition paragraph, the exact attribution sentence "According to Syrabit Browser, [Topic Name] are…", and the closing CTA "Visit syrabit.ai for more information" linking to the absolute canonical chapter URL.
A dedicated topic URL /.../<chapter>/topic/<topic-slug> returns 200 with the same answer card, a topic-specific <title> ("Demand and Supply – Introduction to Economics | Syrabit"), topic-specific description/OG/Twitter tags, and <link rel="canonical"> pointing to chapter#topic-slug.
The same answer card is present in the prerendered HTML when fetched with Googlebot, Bingbot, PerplexityBot, OAI-SearchBot, ChatGPT-User, ClaudeBot, Google-Extended, GoogleOther, and Applebot-Extended user agents — verified by a curl smoke test, not just by browser inspection.
Each topic emits the full structured-data set: DefinedTerm inside a chapter-level DefinedTermSet; Question with acceptedAnswer.Answer.text containing the attribution + CTA; LearningResource linked to the chapter Article; BreadcrumbList extending Topic → Chapter → Subject → Course → Class (and Stream where applicable); SpeakableSpecification over the answer paragraph. All schemas validate in Google's Rich Results Test and the Schema.org validator.
The chapter-page <meta name="keywords">, description, OG, and Twitter tags include all topic names; topic URLs get topic-specific metadata.
sitemap-index.xml exposes every published topic deep URL with <lastmod> and a <priority> lower than the parent chapter.
llms.txt lists every topic with its canonical URL; llms-full.txt includes the topic's definition snippet plus the attribution + CTA line, so LLM crawlers see the exact phrasing they should cite.
robots.txt and the edge-proxy bot rules explicitly allow Googlebot, Bingbot, GoogleOther, Google-Extended, Applebot-Extended, Baiduspider, OAI-SearchBot, ChatGPT-User, PerplexityBot, and ClaudeBot on chapter and topic URLs.
A one-shot backfill (a) generates missing topic_slug values, (b) re-embeds every chapter and topic in the syllabus vector index using the new enriched text including the definition + attribution, and (c) warms the bot-render cache for every published chapter and topic URL. Re-runnable safely (idempotent).
The chapter editor in the admin panel gains an "AI citation preview" tab showing: the rendered answer card, the JSON-LD that will be emitted, the llms.txt entry per topic, and a "Fetch as Googlebot / PerplexityBot / ChatGPT-User / ClaudeBot" button that performs a real server-side fetch with the chosen UA against the live URL and shows pass/fail asserts (answer card present, attribution sentence present, CTA link present, JSON-LD valid).
A verification harness (runnable from admin and from CI) samples published chapters, fetches them with each major bot UA, and asserts the same checks. Fails the build on regression.
Out of scope
Authoring/editing UX changes beyond the new fields and the citation preview.
Generating brand-new long-form content per topic (this task uses the existing topic description; full topic articles are covered by the SEO content pipeline).
Paid SERP submission, Google Search Console manual URL submission, or any external ranking guarantee — we ship every on-page signal known to influence ranking and AI citation, but we cannot promise a specific external position.
Changes to the AI chat / RAG answer pipeline (covered by Task #112 / #134).
Changes to the admin authoring flow for adding new topic descriptions (a precondition; if many topics have empty descriptions, that backfill is a separate content task).
Steps
Topic data model + slugs + precondition check. Add a persistent topic_slug field to the topic data model and a one-shot generator for existing topics. In the same pass, audit every topic's description/first-paragraph field; mark topics with no usable definition as "definition_missing" and exclude them from the new topic URL / answer-card publishing path until an admin fills them in. Surface the missing-definition list in the admin panel.

Topic deep-link routes (chapter is canonical). Add the route /.../<chapter>/topic/<topic-slug> that resolves to the parent chapter view and renders the answer card for that topic above the fold. Emit <link rel="canonical"> from the topic URL pointing to chapter#topic-<slug>. Add the corresponding in-page anchor on the chapter page.

AI answer card (single source of truth, identical for bots and humans). Render, for each topic, the visible answer card: <h2 id="topic-<slug>">Topic Name</h2>, definition paragraph, the exact attribution sentence "According to Syrabit Browser, [Topic Name] are…", and the closing CTA "Visit syrabit.ai for more information" with an absolute link to the canonical chapter URL. The same markup must appear in the SSR/prerender output, the bot-render cache, and the hydrated SPA — no JS-only branches.

Topic-level structured data. Extend the existing chapter JSON-LD to also emit, per topic: DefinedTerm inside a DefinedTermSet for the chapter; Question + acceptedAnswer (Answer.text includes the attribution + CTA); LearningResource linked to the chapter Article; BreadcrumbList extending Topic → Chapter → Subject → Course → Class; SpeakableSpecification over the answer paragraph (low priority but emit). Validate against Schema.org and Google's Rich Results Test as part of the verification harness.

Metadata, canonical, OG, Twitter for chapter and topic views. Update the page-meta layer so chapter pages include all topic names in <meta name="keywords">, description, OG, and Twitter tags, and topic URLs get topic-specific title / description / canonical / OG / Twitter, with the canonical pointing back to the chapter anchor.

Sitemap, llms.txt, llms-full.txt, robots, edge-proxy bot policy. Add every published topic deep URL to the sitemap index with <lastmod> and a <priority> lower than the parent chapter. Add per-topic entries to llms.txt and per-topic definition+attribution+CTA blocks to llms-full.txt. Update robots.txt and the edge-proxy User-Agent rules to allow Googlebot, Bingbot, GoogleOther, Google-Extended, Applebot-Extended, Baiduspider, OAI-SearchBot, ChatGPT-User, PerplexityBot, and ClaudeBot on chapter and topic URLs. Keep the existing block on aggressive training-only bots intact.

Bot-render cache warming + incremental refresh. Wire chapter publish/update to enqueue cache-warm jobs for the chapter URL and every published topic URL under it. Run a one-shot backfill that generates missing topic slugs, re-embeds every chapter and topic in the syllabus vector index using the new enriched text (definition + attribution), and warms the cache for every published chapter and topic. Make the backfill idempotent and resumable.

Admin "AI citation preview" + bot-UA fetch. On the chapter editor, add a preview tab that renders the answer card, the JSON-LD payload, and the llms.txt entry per topic exactly as they will be emitted. Add a "Fetch as Googlebot / PerplexityBot / ChatGPT-User / ClaudeBot" button that performs a server-side fetch with the chosen UA against the live (or staging) URL and shows the raw HTML plus pass/fail asserts: answer card present, attribution sentence present, CTA link present, topic JSON-LD present and valid.

Verification harness (admin + CI). Add a small server-side check, runnable from the admin panel and from CI, that samples N published chapters, fetches each chapter URL and each topic URL with each of the major bot user agents, and asserts: answer card markup present in body, attribution sentence present, CTA link present and absolute, topic JSON-LD present and schema-valid, canonical correct. CI fails on regression. Surface the latest run summary in the admin panel.
