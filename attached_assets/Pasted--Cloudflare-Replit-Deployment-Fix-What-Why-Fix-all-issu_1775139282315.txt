 Cloudflare + Replit Deployment Fix
What & Why
Fix all issues preventing production authentication and API communication when the frontend is deployed on Cloudflare Pages (syrabit.ai) and the backend stays on Replit. Three concrete problems exist: cookies lack a domain attribute so cross-origin auth silently fails, CORS doesn't include the production domains by default, and several frontend pages define their own API base instead of using the centralized one.

Done looks like
Login, signup, token refresh, and logout work correctly when frontend is on Cloudflare Pages and backend is on Replit (cookies flow cross-domain)
CORS accepts requests from the Cloudflare Pages domains in production while still working in local dev
All frontend pages use the single centralized API base from utils/api.jsx — no scattered local definitions
Out of scope
Railway deployment (backend stays on Replit)
Dockerfile or railway.toml changes
DNS or Cloudflare dashboard configuration
CI/CD pipeline setup
Deployment guide documentation
Tasks
Add a COOKIE_DOMAIN environment variable to config.py (default empty for dev, set to production domain when deployed). Apply the domain=COOKIE_DOMAIN parameter to all 7 set_cookie calls in auth.py and admin_auth_users.py, and all 4 delete_cookie calls. Only pass the domain kwarg when COOKIE_DOMAIN is set (so dev mode is unaffected).
Update CORS_ORIGINS default handling in config.py to also include common production origins (https://syrabit.ai, https://www.syrabit.ai) when a PRODUCTION_ORIGINS env var is set, in addition to the existing REPLIT_DOMAINS auto-detection. Keep dev-mode localhost defaults working when neither env var is set.
Replace the local API_BASE / API constant definitions in LearnPage.jsx, SubjectLandingPage.jsx, and usePageTracking.js with imports from the centralized utils/api.jsx. This ensures all pages respect VITE_BACKEND_URL consistently.
