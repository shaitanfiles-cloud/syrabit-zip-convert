203 - Migrate Vector Store: MongoDB → Pinecone
What & Why
Move AHSEC/SEBA chapter embeddings from MongoDB Atlas $vectorSearch to a dedicated Pinecone index (syrabit-ahsec). Pinecone's ANN indexes (HNSW/IVF) deliver sub-50 ms p99 recall at scale vs. Atlas's general-purpose vector layer. MongoDB stays as the source of truth for chapters, sessions, user data, and metadata. The migration uses Pinecone's $1 k free credits, so no immediate cost increase.

Done looks like
A syrabit-ahsec Pinecone index (1024-dim cosine, serverless) exists and contains all embedded chunks from the current MongoDB chunks collection.
The RAG semantic retrieval path (rag.py _fetch_chunks_semantic and mongodb_vector.py) queries Pinecone instead of Atlas $vectorSearch.
The chunk ingestion pipeline (chunk_embedder.py) upserts new embeddings to Pinecone going forward.
The subject router's syllabus vector search is also routed to Pinecone (or kept in Atlas if it uses a separate index — evaluate and document the decision).
PINECONE_KEY and PINECONE_INDEX are the only new required env vars; all other env vars are unchanged.
RAG response quality is validated with at least 5 representative AHSEC/SEBA queries against both backends before cutover.
The old Atlas vector_index and embedding fields in chunks are left in place but are no longer written to (safe archive; can be dropped in a follow-up).
Out of scope
Replacing Pinecone Inference reranking (bge-reranker-v2-m3) — it stays as-is.
Migrating user sessions, transactions, or any non-vector MongoDB collections.
Changing the embedding model (Cohere embed-multilingual-v3.0, 1024-dim stays).
Adding hybrid (keyword + vector) search within Pinecone — hybrid stays in its current form (MongoDB keyword + Pinecone vectors → reranker).
Dropping the embedding field from MongoDB chunks (deferred cleanup task).
Steps
Add Pinecone index configuration and secrets — Register PINECONE_KEY and PINECONE_INDEX (syrabit-ahsec) as environment secrets. Create the Pinecone serverless index (AWS us-east-1, 1024-dim, cosine metric) via the Pinecone SDK in an ensure_pinecone_index() helper, mirroring the existing ensure_vector_index() pattern.

Build PineconeVectorRetriever — Create artifacts/syrabit-backend/retrievers/pinecone_vector.py that exposes the same interface as MongoVectorRetriever (query, upsert, delete). Map MongoDB's chunk_id → Pinecone vector ID; store subject_id, chapter_id, board_id, embedding_model as Pinecone metadata. Use filter on metadata for scoped queries (same semantics as the current Atlas filter field).

Write a one-shot data migration script — Create artifacts/syrabit-backend/scripts/migrate_chunks_to_pinecone.py. It reads all chunks documents that have a non-null embedding field from MongoDB, batches them in groups of 100, and upserts to Pinecone. Include a progress counter and a dry-run flag. Log failures per batch without aborting the full run. This script is run once manually by an operator.

Swap the retriever in rag.py — Replace the $vectorSearch pipeline in _fetch_chunks_semantic (lines ~473-549) with a call to PineconeVectorRetriever.query(...). Preserve the existing two-step pattern: get chunk hits → extract unique chapter_ids → fetch full chapter content from MongoDB. Keep _vs_score naming for the score field so downstream reranking code is unchanged.

Swap the retriever in chunk_embedder.py — After generating embeddings in embed_chunks_bulk, upsert vectors to Pinecone instead of (or in addition to, during a transition window) updating the embedding field in MongoDB. Make this behaviour toggle-able via a PINECONE_WRITE env flag so the old path can be kept warm during validation.

Update MongoVectorRetriever usage across the codebase — Search for any other callers of MongoVectorRetriever or direct $vectorSearch pipelines (e.g., the syllabus subject router). For each: decide whether to route to Pinecone (if it uses the same chunks index) or keep in Atlas (if it uses a separate collection/index). Document the decision in a code comment.

Validate and cut over — Run the migration script against the production chunks collection. Execute 5+ representative AHSEC/SEBA queries using both backends in parallel (shadow mode) and compare top-k chapter IDs and scores. Once parity is confirmed, flip PINECONE_WRITE=true and remove the Atlas vector search path from the hot code path.
