Full-Stack Upgrade, GitHub Push & Production Redeploy
What & Why
Perform a complete, end-to-end production upgrade across all layers of the Syrabit.ai platform: upgrade every dependency to its current stable release, modernise API endpoint patterns in both the FastAPI Python backend and the Rust Axum core, push the resulting clean codebase to GitHub, and then execute a verified production redeploy across both Railway services and all Cloudflare targets (Pages frontend + edge-proxy and email Workers). The goal is a single, atomic "raise the waterline" cycle that leaves no service running on stale packages or mismatched configurations.

Done looks like
Every package.json, pnpm-workspace.yaml catalog, requirements.txt / pyproject.toml, and Cargo.toml is updated to the latest stable release of each dependency with no unresolved peer warnings.
FastAPI and Pydantic v2 patterns are used consistently throughout artifacts/syrabit-backend; all response models are typed; deprecated helpers are removed.
Rust Axum backend compiles cleanly against the latest crate versions; no cargo audit warnings remain.
Cloudflare Worker compatibility dates in both wrangler.toml files are set to the current date and Wrangler CLI is on its latest release.
All changes are committed on the main branch and pushed to GitHub; CI passes (Vitest, Playwright, backend tests, WAF drift check).
Both Railway services (Python backend, Rust core) are redeployed and /healthz/ai, /livez, /readyz, and /health all return 200.
Cloudflare Pages frontend and both Workers (edge-proxy, email-worker) are redeployed and smoke tests pass.
The existing edge-proxy-deploy.yml pipeline auto-rollback guard is exercised — production is only promoted after preview smoke tests pass.
Admin health panel shows all services green after deploy.
Out of scope
Introducing new product features or UI changes.
Migrating to a different database, hosting provider, or infrastructure paradigm.
Adding new API endpoints beyond what is already planned.
Changes to existing open PROPOSED tasks — this task touches only upgrade and deploy concerns.
Steps
Phase 1 — Dependency Audit & Upgrade
Python backend dependency upgrade — Inside artifacts/syrabit-backend, audit requirements.txt / pyproject.toml. Use pip list --outdated and upgrade FastAPI, Pydantic (ensure v2 throughout), Uvicorn, httpx, motor, supabase-py, google-generativeai, groq, and all other first-party deps to their latest stable releases. Pin exact versions in the lockfile.

Rust Cargo upgrade — Inside backend/rust-core, run cargo update and then cargo upgrade (cargo-edit) to bump all crates in Cargo.toml to their latest compatible semver version. Run cargo audit and resolve any advisories. Ensure the project compiles cleanly with cargo build --release.

Node / pnpm catalog upgrade — At the monorepo root, update the shared pnpm-workspace.yaml catalog: bump React (already 19.1.0 → verify no newer patch), Vite (7.x → latest 7.x patch), Vitest, Playwright, Wrangler (4.x → latest), @supabase/supabase-js, @tanstack/react-query, lucide-react, framer-motion, Tailwind CSS v4, TypeScript (~5.9.x → latest), and all @radix-ui packages. Run pnpm install and resolve any peer conflicts. Verify pnpm-lock.yaml is updated.

Root override bump — Update the pnpm.overrides block in root package.json for security-pinned packages (undici, postcss, path-to-regexp, follow-redirects, lodash, picomatch) to their latest patched versions.

Worker-local dependency sync — Inside workers/edge-proxy and workers/email-worker, align their local package.json Wrangler and TypeScript versions to match the catalog, delete stale package-lock.json files, and re-lock with pnpm install from the monorepo root.

Phase 2 — API & Backend Code Modernisation
FastAPI / Pydantic v2 sweep — Audit every router in artifacts/syrabit-backend. Replace any Pydantic v1 .dict() calls with .model_dump(), orm_mode with model_config = ConfigDict(from_attributes=True), and @validator decorators with @field_validator. Add response_model= typing to any admin endpoint that is still untyped. Verify lifespan context manager replaces deprecated on_event startup/shutdown handlers.

Endpoint health & readiness alignment — Ensure /livez returns the application version, Git SHA, and uptime; /readyz checks MongoDB, Supabase, and Redis connectivity before returning 200; /healthz/ai pings the Cloudflare AI Gateway. Document these contracts briefly in RAILWAY-DEPLOY.md.

Rust Axum route modernisation — Update any deprecated Axum 0.7→0.8 handler patterns (e.g., extract::State wiring, IntoResponse impls). Confirm gRPC reflection is enabled for the 50051 port. Confirm WebSocket /ws/metrics handler compiles and reconnects cleanly.

Cloudflare Worker compatibility date update — Set compatibility_date in workers/edge-proxy/wrangler.toml and workers/email-worker/wrangler.toml to today's date (2026-05-01). Review the Cloudflare changelog for any breaking flags introduced since the previous date and fix any affected worker code.

Phase 3 — CI & GitHub Push
Run the full test suite locally — Execute pnpm --filter artifacts/syrabit test, pnpm --filter artifacts/syrabit build, backend pytest suite, and cargo test in Rust core. Fix any failures caused by the version bumps.

Update GitHub Actions workflow dependencies — In all .github/workflows/*.yml files, bump action versions (actions/checkout, actions/setup-python, actions/setup-node, actions/cache) to their latest major releases. Ensure railway-deploy.yml and edge-proxy-deploy.yml reference the current Wrangler version.

Commit and push to GitHub — Stage all changes with a structured commit message (chore: full dependency upgrade + api modernisation + deployment config refresh). Push to the main branch. Confirm all GitHub Actions CI workflows turn green before proceeding to production deploy.

Phase 4 — Railway Redeployment
Redeploy Python FastAPI backend on Railway — Using the Railway CLI (via scripts/railway.sh), trigger a new deployment for the Python backend service. Confirm the correct environment variables are set (Supabase keys, MongoDB URI, Groq/Gemini/Sarvam keys via CF AI Gateway, Upstash Redis URL, Cloudflare R2/KV/D1 tokens). Watch build logs; confirm the service passes its health probe before marking complete.

Redeploy Rust Axum core on Railway — Trigger a new deployment for the Rust core service from backend/rust-core/railway.toml. Confirm gRPC port 50051 and HTTP REST port are both bound. Verify /health returns 200 and D1 sync endpoints respond correctly.

Post-Railway smoke test — Hit /livez, /readyz, /healthz/ai, /api/edge/d1-status, and /health against the live Railway URLs. Confirm all return expected JSON with no 5xx errors.

Phase 5 — Cloudflare Redeployment
Deploy edge-proxy Worker — Run the edge-proxy-deploy.yml pipeline (or wrangler deploy from workers/edge-proxy). The pipeline deploys to preview first, runs automated smoke tests from monitored-urls.json, requires manual approval gate, then promotes to production. Confirm auto-rollback does not trigger.

Deploy email-worker — Run wrangler deploy from workers/email-worker. Verify the worker bindings (D1, KV, R2, secrets) are correctly resolved in the production environment.

Deploy frontend to Cloudflare Pages — Run pnpm --filter artifacts/syrabit build then trigger a Cloudflare Pages deployment (via wrangler pages deploy or the Pages Git integration). Confirm _routes.json and _headers files are picked up correctly and the Pages Worker is active. Verify the production URL returns the React app with no CSP or CORS errors.

Post-Cloudflare smoke test — Using workers/edge-proxy/monitored-urls.json, verify all monitored routes return expected status codes. Check the admin health panel for green status across all services. Confirm Cloudflare Cache, WAF, and mTLS indicators are armed.

Phase 6 — Post-Deploy Validation & Cleanup
Admin health panel sign-off — Open the admin health panel and confirm: Railway Python backend green, Railway Rust core green, Cloudflare Pages green, edge-proxy green, email-worker green, AI Gateway green, MongoDB connected, Supabase connected, Redis connected.

Secondary AI key removal from Railway — As called for in Task #177, now that the BYOK setup is verified live, remove any redundant secondary AI provider keys from Railway environment variables, leaving only the CF AI Gateway credentials.

Update deployment documentation — Briefly update RAILWAY-DEPLOY.md, CLOUDFLARE_PAGES.md, and PHASE3_DEPLOYMENT_GUIDE.md to reflect the new package versions, wrangler compatibility date, and any configuration changes made during this cycle.
