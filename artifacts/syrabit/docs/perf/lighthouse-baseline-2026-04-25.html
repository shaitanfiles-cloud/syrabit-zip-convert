Note on report format. A real Lighthouse run requires a published deployment (so the change must ship to syrabit-zip-convert.pages.dev first) and a headless Chromium with the Lighthouse CLI installed in the runtime that captures the report. The Replit build environment has neither permanently available, so this report is a structural audit of the built dist/ tree captured against the same 15 SPA-fallback / prerendered HTMLs. The metrics below — render-blocking CSS bytes, inlined critical-CSS bytes per route — directly drive Lighthouse's "Eliminate render-blocking resources" and "Reduce unused CSS" opportunities, which were the two PageSpeed audit failures called out in the task brief.

Every prerendered + SPA-fallback HTML loads the 141.4 KB main stylesheet via a synchronous <link rel="stylesheet"> in <head>. The browser cannot draw the first frame until that file finishes downloading and parsing. On a typical AHSEC/SEBA student's 4G CGNAT connection, this adds 200–500 ms to FCP and proportionally to LCP for hero text/imagery.

Methodology

  • Walks every *.html under artifacts/syrabit/dist/ (excluding the GSC verification stub, the rollup-visualizer report, and the SW offline shell, which never reference the main stylesheet).
  • Render-blocking = a top-level <link rel="stylesheet" href="/assets/index-…css"> with no onload swap and no preload promotion. The <noscript> fallback link does not count, since it only activates with JavaScript disabled.
  • Inline <style> = sum of bytes between every <style>…</style> pair in the document head and body. Includes both pre-existing inline blocks (Emergent badge hider, pre-hydration shell) and any extracted critical-CSS payload.
